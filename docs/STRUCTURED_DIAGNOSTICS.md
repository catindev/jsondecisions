# Канон structured diagnostics в `jsondecisions`

Этот документ фиксирует единый паттерн структурированных диагностических сообщений для библиотеки `jsondecisions`, будущего `jsondecisions-cli` и соседних модулей вроде `jsonmapping`.

## Цель

Structured diagnostics нужны затем, чтобы ошибки можно было:

- показывать человеку в CLI и UI;
- связывать с конкретным артефактом и местом в нём;
- обогащать путём к файлу через `sources`;
- стабильно тестировать без парсинга строк регулярными выражениями.

## Базовая форма

Каждая compile-time ошибка должна иметь форму:

```js
{
  code: 'UNRESOLVED_REF',
  artifactId: 'entrypoints.beneficiary.register.find_create',
  path: 'rules[2]',
  message: 'resolves to "..." which does not exist'
}
```

Поля:

- `code` — машинно-читаемый код ошибки;
- `artifactId` — идентификатор артефакта, внутри которого обнаружена ошибка;
- `path` — максимально узкий путь до проблемного места внутри артефакта;
- `message` — человекочитаемое описание.

## Принцип узкого `path`

Если ошибка относится к конкретному полю внутри артефакта, `path` должен указывать **максимально узко**, а не на ближайший общий контейнер.

Хорошо:

- `when.abs.findResult`
- `then.tags[1]`
- `defaultDecision.reason`
- `requiredFacts[3]`
- `rules[2]`

Плохо:

- просто `when`
- просто `then`
- просто `rules`

Причина проста: CLI и будущий UI должны уметь показать, **что именно** сломано, не заставляя пользователя читать весь JSON целиком.

## Обязанности полей

### `code`

`code` отвечает на вопрос: **что за тип ошибки произошёл**.

Примеры:

- `MISSING_ID`
- `UNKNOWN_ARTIFACT_TYPE`
- `INVALID_WHEN_VALUE_TYPE`
- `UNRESOLVED_REF`
- `REF_TARGET_WRONG_TYPE`

`code` должен быть стабильным и пригодным для ветвления в CLI.

### `artifactId`

`artifactId` отвечает на вопрос: **в каком артефакте ошибка**.

Это значение позволяет CLI найти файл через `sources.get(artifactId)?.file` и показать пользователю физический путь к JSON-файлу.

### `path`

`path` отвечает на вопрос: **в каком месте артефакта ошибка**.

Это не JSONPath общего назначения, а простой диагностический путь, понятный человеку.

### `message`

`message` отвечает на вопрос: **что именно не так**.

Это поле пишется для человека и не должно использоваться как источник истины для логики CLI.

## Связь с `sources`

`compile()` уже принимает `sources: Map<string, SourceInfo>`.

Канонический способ использования в CLI:

1. получить `CompilationError.errors`;
2. взять `artifactId` из каждого entry;
3. попробовать найти файл через `sources.get(artifactId)`;
4. вывести:
   - код ошибки;
   - артефакт;
   - путь внутри артефакта;
   - файл;
   - сообщение.

## Что не надо делать

Не надо:

- парсить строки ошибок регулярками;
- сплющивать `path` до уровня общего контейнера;
- использовать `message` как machine-readable поле;
- терять `artifactId` ради краткости.

## Ориентир для будущих модулей

Этот же паттерн рекомендуется использовать в:

- `jsondecisions-cli`
- `jsonmapping`
- возможном `jsonmapping-cli`

То есть единый канон такой:

> если ошибка относится к конкретному месту внутри артефакта, `path` должен указывать максимально узко, `artifactId` должен указывать, где именно это произошло, а `message` должен оставаться человекочитаемым объяснением, а не суррогатом структуры.
